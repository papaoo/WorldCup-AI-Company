# 世界杯 AI 公司产品重构审查

日期：2026-06-09  
范围：`PiPiClaw.Team` 后端、当前 React 前端、世界杯预测业务闭环、下一版 UI 信息架构

## 1. 结论先行

当前系统已经具备可复用的底层能力：球队对象、员工绑定、公开数据快照、基线预测、LLM 调用记录、证据包、日志审计、生产/测试数据隔离。它不是失败的原型。

但当前产品体验不成立，原因不是单纯“UI 丑”，而是信息架构错了：界面把 workflow、artifact、harness、snapshot、日志等工程对象暴露给普通用户，导致用户第一眼看不到“这场比赛谁更可能赢、依据是什么、我该不该信”。

下一版必须把主线收敛为：

```text
比赛 -> 双方球队 -> 数据证据 -> 胜平负概率 -> 风险与不确定性 -> CEO 最终研判 -> 可追溯审计
```

“AI 公司/员工”应该是产品表现层和协作隐喻，不应该压过赛事预测本身。

## 2. 当前系统真实状态

已确认的生产视图：

- 球队：48 支。
- 已导入小组赛：72 场。
- 当前生产接口默认不混入 demo/harness 数据。
- 基线预测已可为 72 场小组赛生成。
- LLM 调用记录、成本估算、成功/失败状态已经可追踪。
- 数据源质量审计已经能输出来源、可靠度、是否需要 key、适用范围。

必须承认的限制：

- 2026 世界杯完整赛制是 48 队、12 个小组、总计 104 场；当前系统只落了小组赛 72 场，淘汰赛 32 场需要随结果生成或按占位赛程建模。
- 目前大量球队的 FIFA rank 仍是占位值 50，导致很多比赛出现 37% / 26% / 37% 的平衡预测。
- RSS 新闻源只能作为线索发现，不能直接当成已确认事实。
- 当前 baseline 是“可运行的策略骨架”，还不是高精度预测模型。
- UI 中文有乱码，必须作为阻断级问题修复。

## 3. 数据可信度设计

用户最终要相信预测，不是因为界面好看，而是因为每个结论都能回答四个问题：

- 数据从哪里来。
- 数据是否稳定。
- 模型用了哪些字段。
- 哪些地方仍然不确定。

建议把数据分为四层：

| 层级 | 用途 | 示例 | UI 展示 |
|---|---|---|---|
| 官方参考层 | 赛程、场馆、赛制校验 | FIFA 官方赛程页 | 显示为“官方参考” |
| 结构化赛程层 | 自动采集、去重、入库 | FixtureDownload、openfootball、worldcup26 public feed | 显示为“已交叉校验” |
| 情报线索层 | 新闻、伤病、舆情候选 | ESPN/Guardian RSS | 显示为“待分拣线索” |
| 模型解释层 | 总结、风险、报告 | DeepSeek/PiPiClaw LLM | 显示为“模型审查意见” |

重要原则：

- 官方参考用于校验事实，不一定批量抓取。
- 结构化源用于自动采集。
- 新闻源先进入情报队列，不直接改概率。
- LLM 负责提取、压缩、解释和质检，不负责创造事实。

## 4. 应保留的能力

- `WatchObject = 球队` 的抽象，未来可以复用到股票、竞品、客户。
- 一个球队绑定一个长期 AI 员工。
- 自动采集数据源、生成快照、去重。
- 情报分拣和变化检测。
- baseline 策略层，必须独立于 LLM。
- LLM 报告、风险审查、CEO 汇总。
- 日志审计和成本统计。
- harness，但只给开发/审计入口，不进入主界面。

## 5. 应隐藏或删除的用户界面复杂度

从普通用户主界面移除：

- workflow run 列表。
- artifact 列表。
- harness 按钮。
- 原始 snapshot 表格。
- 内部异常堆栈。
- token 明细长表。
- “演示数据/真实数据/运行闭环”这类工程按钮。

保留在角落或高级抽屉：

- 数据源审计。
- LLM 成本。
- 最近日志。
- 失败原因。
- 回测报告。
- 原始证据链接。

## 6. 推荐的新前端信息架构

### 一级页面

1. 赛事预测中枢  
   默认首页。按日期和小组展示比赛，点击比赛进入分析。

2. 比赛详情  
   一场比赛一个完整判断页：胜平负概率、关键证据、风险、员工意见、CEO 结论。

3. 球队研究室  
   一个球队一个员工：球队档案、近期赛程、情报线索、历史报告、数据质量。

4. 数据可信度  
   数据源状态、最近采集、来源快照、质量门禁。

5. 审计抽屉  
   开发者/高级用户查看 workflow、artifact、token、错误、harness。

### 一场比赛详情页的建议模块

```text
比赛标题
  - 中文球队名、组别、开球时间、场馆

预测结论
  - 主胜 / 平 / 客胜
  - 置信度
  - 风险等级
  - 最终一句话

证据包
  - 赛程来源
  - 球队排名/实力
  - 新闻线索
  - 伤病/阵容状态
  - 市场赔率（未来可选）

员工协作
  - 主队研究员
  - 客队研究员
  - 数据分析师
  - 风险官
  - CEO

审计
  - 策略版本
  - 参与快照
  - LLM 调用
  - 成本
  - 生成时间
```

## 7. 三个 UI 方向

### 方向 A：预测中枢

适合正式上线。  
气质：可信、冷静、高效、数据研判。  
用户第一眼看到赛程、胜率和证据质量。员工隐喻弱一些，主要作为“谁负责分析”的身份出现。

优点：

- 最容易做成生产级产品。
- 信息密度高但不乱。
- 适合后续加入股票、竞品等场景。

风险：

- 如果视觉素材不足，容易退化成普通 dashboard。

### 方向 B：战术桌

适合形成差异化。  
气质：战术板、球探桌、赛事研究所。  
中间是可交互战术桌，周围是员工、证据、概率和风险。

优点：

- 与世界杯主题贴合。
- 比表格更有记忆点。
- 适合动画化：证据卡飞入、概率曲线变化、员工发言。

风险：

- 需要更强的布局控制，否则会显得玩具化。

### 方向 C：赛事演播室

适合展示报告和结论。  
气质：电视演播室、赛前节目、CEO briefing。  
一屏聚焦一场比赛，像播报一样输出判断。

优点：

- 最容易让用户读懂“结论是什么”。
- 适合后续生成日报、分享图、公众号文章、PPT。
- 可以保留 AI 员工角色感。

风险：

- 批量浏览 72 场时效率不如方向 A。

## 8. 下一版后端接口建议

当前前端一次性拉太多底层对象。建议新增 BFF 接口，直接服务新 UI：

```text
GET /api/worldcup/product/overview
```

返回：

- 当前阶段。
- 数据健康。
- 今日/近期比赛。
- 高关注比赛。
- 数据源摘要。
- 成本摘要。

```text
GET /api/worldcup/product/matches
```

返回比赛卡片列表：

- match_id
- group_name
- kickoff_time
- venue
- home_team / away_team 中文名、代码、旗帜、员工
- probabilities
- confidence
- risk_level
- data_quality
- latest_summary

```text
GET /api/worldcup/product/matches/{id}
```

返回一场比赛详情：

- 比赛基本信息。
- 预测概率。
- 策略因素。
- 证据包。
- 员工报告摘要。
- CEO 结论。
- 审计摘要。

```text
POST /api/worldcup/product/matches/{id}/refresh
```

用户主动刷新一场比赛：

- 采集相关球队数据。
- 分拣变化。
- 必要时调用 LLM。
- 更新预测与报告。

```text
GET /api/worldcup/product/teams/{id}
```

返回球队研究室：

- 球队档案。
- 员工信息。
- 赛程。
- 情报线索。
- 最新报告。
- 数据质量。

## 9. 预测模型下一步

不要让 LLM 直接给胜率。更稳的架构是：

```text
结构化数据 -> 策略模型计算概率 -> LLM 审查解释 -> 风险官降级/提示 -> CEO 汇总
```

策略层下一步需要加入：

- 真实 FIFA 排名或 Elo/SPI 类强度评分。
- 最近战绩。
- 进失球表现。
- 对手强度修正。
- 主场/旅行/气候。
- 伤病与阵容确认程度。
- 新闻情绪只作为低权重辅助。
- 赔率作为未来可选高价值校准源。

必须做回测：

- 命中率。
- Brier Score。
- Log Loss。
- 与简单基线比较。
- 每次策略版本变更必须保留结果。

## 10. UI 开发约束

- 中文优先，英文只保留必要缩写。
- 默认页面不出现 harness、artifact、workflow 等工程词。
- 每个概率旁边必须有置信度和数据质量。
- 每个强结论必须能展开证据。
- 所有按钮必须表达用户目的，例如“刷新这场比赛”“生成赛前研报”，不要写“运行闭环”。
- 预测不是投注建议，必须保留边界说明。
- 动画和素材服务于理解，不做无意义装饰。

## 11. 当前优先级

第一优先级：

- 修复中文乱码。
- 新增产品级 BFF 接口。
- 首页改为“比赛预测中枢”。
- 比赛详情页按证据链组织。
- 所有假数据/占位排名显式标记，不能伪装成真实强预测。

第二优先级：

- 加入真实球队强度数据。
- 建立策略回测。
- 情报变化触发单场刷新。
- 生成可信研报。

第三优先级：

- 游戏风格 UI 动画。
- Q 版员工素材系统。
- 分享图、PPT、Word、移动端。

## 12. 我的建议

产品路线选方向 A 作为生产骨架，吸收方向 B 的战术桌视觉，再用方向 C 做报告输出页。

也就是说：

```text
首页 = 预测中枢
比赛详情 = 战术桌
报告/分享 = 赛事演播室
审计 = 角落抽屉
```

这条路线最稳：既能把系统做清楚，也能逐步变得有风格，不会为了动画牺牲可信度。
